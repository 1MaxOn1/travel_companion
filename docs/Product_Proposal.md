# Product Proposal

## 1. Обоснование идеи

**Прикладная задача**  
Путешественник хочет спланировать поездку, которая соответствует его бюджету, предпочтениям и ограничениям (визы, сезонность, особые потребности). Сегодня этот процесс — ручная работа с 5–10 сервисами: поиск билетов, отелей, карты, отзывы, погода, визовые калькуляторы. Нет единого места, где хранились бы предпочтения, а при изменении планов всё приходится перестраивать заново.

**Почему агентный подход**  
Задача требует не просто поиска данных, а последовательного рассуждения: уточнить неполный запрос, проверить совместимость ограничений (бюджет + виза + сезон), предложить компромиссы, адаптировать план при изменении условий. Простая LLM без orchestration не сможет надёжно вызывать несколько внешних API, вести сквозную память о предпочтениях и поддерживать диалог с пересчётом бюджета. Только агентная система с состоянием и набором специализированных инструментов способна эмулировать поведение персонального ассистента.

---

## 2. Цель и метрики

**Цель проекта**  
Разработать PoC агентной системы, которая через чат-интерфейс принимает свободный запрос о путешествии, уточняет недостающие данные, собирает информацию из эмулируемых источников (погода, визы, билеты, отели, события), формирует персонализированный маршрут с учётом сквозной памяти и генерирует итоговый отчёт с бюджетом и чек-листом за время, сопоставимое с общением с живым ассистентом.

**Метрики успеха**

*Продуктовые метрики*

| Метрика | Цель |
|--------|------|
| Время от первого сообщения до итогового отчёта | ≤ 3 мин (p95) |
| Доля успешно завершённых сессий (без ручного прерывания) | ≥ 85% |
| Пользовательская оценка полезности рекомендаций (по опроснику) | ≥ 4.0 из 5 |

*Агентные метрики*

| Метрика | Цель |
|--------|------|
| Точность определения ограничений (виза, сезон, диеты) из запроса | ≥ 95% |
| Корректность формирования маршрута (дни, логистика) | ≥ 90% |
| Hallucination rate в итоговом отчёте | < 2% |

*Технические метрики*

| Метрика | Цель |
|--------|------|
| p95 latency полного пайплайна (до отчёта) | ≤ 180 сек |
| Доля успешных tool calls (эмуляция API) | ≥ 95% |
| Стоимость LLM-токенов на одну сессию | ≤ $0.15 |

## 3. Сценарии использования и edge-кейсы

**Основные сценарии**

*Сценарий 1 – Базовый запрос с уточнением*  
Пользователь пишет: *«Хочу съездить куда-нибудь тёплое на 5–7 дней в июне, бюджет до 1000 евро на двоих, без визы, любим историю и вино»*.  
Агент проверяет бюджетные ограничения, предлагает Турцию или Болгарию, уточняет предпочтения по типу жилья, после выбора строит план по дням с экскурсиями, учитывая, что пользователь ранее указал «не ест свинину» (извлечено из памяти).

*Сценарий 2 – Изменение планов*  
Пользователь после получения маршрута просит: *«Давай удешевим»*. Агент предлагает альтернативные даты, отели классом ниже или перелёт с пересадкой, пересчитывает бюджет и показывает новую версию.

*Сценарий 3 – Сложные ограничения*  
Семья с ребёнком 3 лет запрашивает поездку в Италию в августе. Агент обнаруживает, что на выбранные даты в регионе сильная жара (погодное API) и начинается религиозный праздник, из-за чего закрыты многие музеи. Агент предупреждает и предлагает сместить даты на сентябрь или выбрать другое направление.

**Edge-кейсы**

| Edge-кейс | Ожидаемое поведение агента |
|-----------|----------------------------|
| Запрос слишком общий (например, «хочу отдохнуть») | Запрашивает уточнения: бюджет, даты, интересы, состав путешественников |
| Бюджет не соответствует выбранному направлению | Предлагает компромиссные варианты (другая страна, смена сезона, класс обслуживания) |
| Визовые требования не учтены пользователем | Явно подсвечивает «красный флаг»: «Для поездки в X нужна виза, оформление минимум 2 недели» |
| Пользователь указывает даты, когда в регионе сезон дождей | Предупреждает и предлагает альтернативные даты или направления |
| Эмулируемый API погоды недоступен | Использует кэшированные данные с указанием, что информация может быть неактуальной |
| Память о предпочтениях противоречит новому запросу | Запрашивает подтверждение: «Вы ранее указывали, что предпочитаете прямые рейсы, но сейчас они не вписываются в бюджет. Разрешить пересадку?» |

## 4. Ограничения

**SLO**

| Параметр | Значение |
|----------|---------|
| p95 latency (до итогового отчёта) | ≤ 180 сек |
| p99 latency | ≤ 240 сек |
| Таймаут одного tool call (эмуляция) | 5 сек, 2 retry |
| Количество инструментов, вызываемых за сессию | ≤ 20 |

**Операционные ограничения**

| Параметр | Оценка |
|----------|--------|
| Бюджет на LLM на сессию | ≤ $0.15 |
| Инфраструктура | Веб-чат (для демо), деплой на сервере |
| Обновление данных | Эмуляция на тестовых датасетах; в PoC нет real-time интеграций |
| Покрытие направлений | 5–10 популярных стран (Турция, Италия, Испания, Таиланд, ОАЭ и др.) |
| Регуляторные ограничения | Явное указание, что система не осуществляет бронирование и не несёт ответственности за точность эмулируемых данных |

## 5. Архитектурный набросок

**Модули и интеграции**

- **Orchestrator Agent (Main Agent)** – ведёт диалог, управляет состоянием, определяет, когда нужно вызвать инструменты.  
  Инструменты: `clarify_intent`, `get_user_preferences`, `update_plan`.

- **Memory Module** – хранит сквозные предпочтения (диеты, типы перелётов, любимые отели) в рамках сессии и между сессиями (persistent storage для демо).  
  Инструменты: `save_preference`, `load_preferences`.

- **Weather Agent** – эмулирует API погоды. Возвращает среднюю температуру, сезон дождей, комфортность для выбранных дат.  
  Инструменты: `get_weather_forecast`.

- **Visa Agent** – проверяет необходимость визы, сроки оформления, возможные сложности.  
  Инструменты: `check_visa_requirements`.

- **Flight Agent** – эмулирует поиск авиабилетов с фильтрацией (прямые рейсы, бюджет).  
  Инструменты: `search_flights`.

- **Hotel Agent** – эмулирует поиск отелей с фильтрацией (бассейн, завтрак, семейные номера, доступность для инвалидов).  
  Инструменты: `search_hotels`.

- **Attraction Agent** – база достопримечательностей с тегами (история, вино, детские развлечения).  
  Инструменты: `get_attractions`.

- **Event Agent** – эмулирует API событий (фестивали, концерты, праздники).  
  Инструменты: `get_events`.

- **Report Agent** – формирует итоговый отчёт: сводка по перелётам, отель, план по дням, бюджет с разбивкой, чек-лист.  
  Инструменты: `generate_report`.

**State** – хранит в рамках сессии:  
- исходный запрос и уточнения  
- выбранное направление и даты  
- предпочтения из памяти  
- собранные данные (погода, виза, варианты перелётов, отели, достопримечательности)  
- текущий бюджет и предложенные компромиссы  
- итоговый маршрут и отчёт

## 6. Data Flow и распределение ответственности

1. **Пользователь отправляет сообщение** → Orchestrator Agent  
2. Orchestrator проверяет, достаточно ли данных для начала планирования:  
   - если нет, вызывает `clarify_intent` (уточняет даты, бюджет, интересы)  
   - если есть, загружает сквозные предпочтения из Memory Module  
3. Orchestrator запускает параллельные вызовы инструментов:  
   - Weather Agent (`get_weather_forecast`)  
   - Visa Agent (`check_visa_requirements`)  
   - Flight Agent (`search_flights`)  
   - Hotel Agent (`search_hotels`)  
   - Attraction Agent (`get_attractions`) – по интересам  
   - Event Agent (`get_events`) – для выбранных дат  
4. Полученные данные агрегируются в State. Orchestrator выполняет reasoning:  
   - проверяет, нет ли «красных флагов» (виза, сезон дождей, превышение бюджета)  
   - если есть противоречия, предлагает компромиссы через диалог  
   - иначе формирует маршрут по дням (вызывает логику ранжирования достопримечательностей)  
5. После утверждения пользователем маршрута Orchestrator передаёт управление Report Agent, который генерирует итоговый отчёт со сводкой и чек-листом.  
6. Отчёт отправляется пользователю.  
7. Все новые предпочтения (например, «пользователь согласился на пересадку») сохраняются в Memory Module для будущих сессий.
