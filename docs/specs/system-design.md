# Системный дизайн Travel Companion

## 1. Ключевые архитектурные решения

| Решение | Обоснование |
|---------|--------------|
| **Mistral AI через OpenAI‑совместимый API** | Низкая стоимость, хорошее качество для русского языка, поддержка function calling (tools). |
| **Один инструмент `search_internet`** | Минимализм — туристическому ассистенту нужен только поиск актуальных данных. Легко расширяется через список `TOOLS`. |
| **Telegram бот (aiogram 3.x + FSM)** | Простой интерфейс, встроенная машина состояний для пошагового сбора параметров (destination, dates, budget, interests). |
| **Синхронный вызов LLM внутри асинхронного хендлера** | Mistral API отвечает за 2–5 секунд, блокировка не критична для демо/бота с низкой нагрузкой. |
| **Отсутствие постоянной памяти (stateless)** | `UserMemory` заготовлен, но не используется. Все данные сессии живут только в FSM и удаляются после генерации плана. |

## 3. Workflow полного диалога

1. **Пользователь** отправляет `/start`
2. **Бот** → запрашивает направление → FSM переходит в состояние `destination`
3. **Пользователь** → вводит город/страну → сохраняем в FSM (`destination`)
4. **Бот** → запрашивает даты → состояние `dates`
5. **Пользователь** → вводит даты → сохраняем
6. **Бот** → запрашивает бюджет (inline-кнопки) → состояние `budget`
7. **Пользователь** → выбирает бюджет → сохраняем
8. **Бот** → запрашивает интересы → состояние `interests`
9. **Пользователь** → вводит интересы → состояние `generating`
10. **Бот** → формирует системный промпт (содержащий все собранные данные и инструкцию) → вызывает `LLMTravelAgent.process_message()`
11. **Агент (llm.py)** выполняет цикл:
    - Отправляет запрос Mistral с инструментом `search_internet`
    - Модель решает, какие поисковые запросы сделать
    - `search_internet` обращается к DuckDuckGo (5 результатов)
    - Результаты возвращаются модели как `tool` сообщение
    - Повторяет до тех пор, пока модель не выдаст финальный ответ или не будет превышен лимит итераций (5)
12. **Бот** → получает финальный ответ → разбивает на чанки по 4000 символов (ограничение Telegram) → отправляет пользователю
13. **FSM** → очищается (`state.clear()`)

## 4. State, Memory, Context

### FSM (aiogram Finite State Machine)
- Состояния: `Form.destination`, `Form.dates`, `Form.budget`, `Form.interests`, `Form.generating`
- Данные: словарь `{"destination": str, "dates": str, "budget": str, "interests": str}`

### UserMemory (задел)
```python
class UserMemory:
    def __init__(self, user_id: int):
        self.user_id = user_id
        self.preferences = {}
    def update(self, key, value): ...
    def get(self, key, default=None): ...
```

###Контекст LLM
- Проблема: история диалога не сохраняется между вызовами process_message(). Каждый вызов начинается с чистого messages = [system, user].
- Последствие: агент не помнит предыдущие вопросы пользователя в рамках одной сессии (например, "а что насчёт отелей?" после первого плана).
- Решение: хранить историю сообщений в UserMemory или в FSM и передавать её в process_message параметром.

## 5. Retrieval‑контур (как агент получает информацию)

Пользовательский запрос (сформированный промпт)
          ↓
Mistral анализирует и решает вызвать search_internet
          ↓
DuckDuckGo возвращает 5 результатов (title, link, snippet)
          ↓
Mistral интерпретирует результаты, может сделать ещё вызовы
          ↓
Генерирует финальный ответ со ссылками

**Особенности:**
- Отсутствует векторная база данных (RAG). Вся свежая информация — через поиск.
- Лимит итераций = 5, то есть максимум 5 раундов поиска на один запрос.
- Промпт явно требует от модели **реальных ссылок**, что снижает галлюцинации.

## 7. Failure modes (режимы отказов)

| Сбой | Наблюдаемое поведение | Рекомендация |
|------|----------------------|--------------|
| Mistral API не отвечает / невалидный ключ | Исключение → бот падает, пользователь получает ошибку Telegram | Добавить `try-except` в `process_message`, вернуть сообщение «сервис временно недоступен» |
| Таймаут DuckDuckGo | Нет таймаута — запрос может висеть вечно (блокировка бота) | Добавить параметр `timeout` в вызов `DDGS` или использовать `asyncio.wait_for` |
| Пустой поисковый запрос | `search_internet` возвращает `{"error": "Empty query"}` → модель получит ошибку и сообщит пользователю | OK (корректное поведение) |
| Слишком длинный ответ LLM | Разбивается на чанки по 4000 символов, отправляется по частям | OK |
| Превышение 5 итераций цикла | Цикл прерывается принудительно, возвращается последний ответ (может быть без `tool_calls`) | Можно увеличить лимит до 10 или логировать такие случаи |
| Пустой `assistant_message.content` | Возвращается строка "Извините, не удалось сформировать ответ." | OK (защита от `None`) |

## 8. Guardrails (защитные механизмы)

| Механизм | Реализация |
|----------|-------------|
| Максимум вызовов инструментов | Цикл `for _ in range(5)` в `llm.py` |
| Защита от пустого ответа LLM | `return assistant_message.content or "Извините, не удалось сформировать ответ."` |
| Ограничение длины сообщения Telegram | Разбивка по 4000 символов в `handlers.py` |
| Проверка на пустой `query` в `search_internet` | `if not query: return {"error": "Empty query"}` |
| Отсутствие бесконечного цикла | Обеспечено `for _ in range(5)` (даже если модель постоянно вызывает инструменты) |

**Планируемые улучшения:**
- Ограничение длины поискового запроса (макс 200 символов)
- Список стоп-слов (чтобы агент не искал опасный контент)
- Валидация дат и бюджета в FSM (форматы, разумные значения)
- Логирование всех вызовов инструментов для отладки (см. `observability.md`)
